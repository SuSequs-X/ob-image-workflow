# image-workflow

面向 Obsidian Markdown 写作场景的图片工作流插件。它的目标不是做“附件总管”，而是把笔记写作中最常见的图片问题压缩到一个闭环中：**粘贴后命名、尺寸控制、图片体检、批量排版、顺序重排、附件归档、安全清洗、日志回滚**。

> 核心机制：插件围绕 Markdown 图片嵌入链接工作，优先处理 `![[image.png]]`、`![[image.png|500]]`、`![](image.png)` 等图片引用，并通过预览、校验、日志和撤销降低断链风险。

---

## 适用场景

适合以下工作流：

- 写作时频繁粘贴截图、实验图、示意图；
- 希望图片文件名自动带有笔记语境；
- 希望统一图片显示宽度，例如全部设为 `500`；
- 希望检查当前笔记是否存在缺失图片、无尺寸图片、重复引用、同名风险；
- 希望按正文顺序重排图片编号，但不破坏原有语义名称；
- 希望把当前笔记引用的图片归档到同名 `.assets` 文件夹；
- 希望清洗图片路径，但不想因为同名文件导致误指向；
- 希望批量操作后可以撤销。

不建议将本插件扩展为视频、音频、PDF、Canvas、Excalidraw 等所有附件的统一管理器。当前版本的稳定边界是：**Markdown 笔记中的图片嵌入链接**。

## 新增痛点优化

### 1. 当前笔记图片体检

命令：

```text
检查当前笔记图片状态
```

该命令会扫描当前笔记中的图片嵌入，并输出以下指标：

| 指标 | 含义 |
|---|---|
| 图片嵌入 | 当前笔记检测到的图片引用总数 |
| 正常链接 | 可解析且无明显风险的图片链接 |
| 缺失文件 | 链接无法解析到真实图片文件 |
| 无尺寸图片 | 没有 `|500` 或类似尺寸后缀的图片 |
| 重复引用 | 同一张图片在当前笔记中被多次嵌入 |
| 同名风险 | 仓库中存在同名图片，短链接清洗可能误指向 |
| 外部链接 | `http`、`https`、`data:` 等外部图片 |
| 可安全清洗 | 可从路径链接安全收缩为文件名链接的图片 |

体检弹窗提供快捷动作：

- **统一无尺寸图片**；
- **预览安全清洗**；
- **重排图片编号**。

> 系统性风险：体检只基于当前 Obsidian 能解析到的链接状态。如果其他插件或同步盘正在异步修改文件，短时间内可能出现缓存未更新。高风险操作前建议重新执行一次体检。

### 2. 当前笔记图片尺寸统一

命令：

```text
统一当前笔记图片尺寸
```

支持四种范围：

| 范围 | 作用 |
|---|---|
| 仅无尺寸图片 | 只给没有尺寸的图片补充尺寸，推荐默认使用 |
| 全部图片 | 将当前笔记所有图片统一改为同一尺寸 |
| 仅当前标题下图片 | 只处理光标所在标题区块内的图片 |
| 清除全部尺寸 | 删除当前笔记图片链接中的尺寸后缀 |

示例：

```md
![[a.png]]
```

输入 `500` 后变为：

```md
![[a.png|500]]
```

已有尺寸会被覆盖：

```md
![[a.png|300]]
```

变为：

```md
![[a.png|500]]
```

该功能只修改 Markdown 正文，不重命名图片文件，风险低于文件移动和重命名。

---

### 3. 粘贴图片快速面板与尺寸预设

粘贴图片后，插件会显示图片设置面板。现在面板增加了尺寸预设按钮，例如：

```text
300 / 400 / 500 / 600 / 800
```

你可以在设置页中修改预设：

```text
尺寸快捷预设：300,400,500,600,800
```

目标是减少重复输入尺寸的操作成本。

> 前提条件：该功能依赖 Obsidian 的文件创建事件。图片粘贴后如果被其他插件立刻移动或重命名，本插件可能无法接管该文件。

### 4. 命名模板系统增强

设置项：

```text
图片命名模板
```

支持变量：

| 变量 | 含义 |
|---|---|
| `{{fileName}}` | 当前笔记文件名，不含扩展名 |
| `{{note}}` | 当前笔记文件名，等同于 `{{fileName}}` |
| `{{heading}}` | 当前光标所在标题 |
| `{{firstHeading}}` | 当前笔记第一个一级标题 |
| `{{folder}}` | 当前笔记所在文件夹名 |
| `{{dirName}}` | 当前笔记所在文件夹名，等同于 `{{folder}}` |
| `{{index}}` | 当前图片序号，按当前笔记图片数量推算 |

> `{{index}}` 默认不补零，例如 `1`、`2`、`3`。如果需要 `01`，可在设置页手动把“编号位数”改为 `2`。

### 序号冲突修复说明

当图片命名模板包含 `{{index}}` 时，插件现在会优先把末尾序号递增，而不是在完整文件名后再追加重复编号。

例如，当前已存在：

```md
![[张三-测试1-1.png]]
```

下一张图片会生成：

```md
![[张三-测试1-2.png]]
```

不会再生成：

```md
![[张三-测试1-1-1.png]]
```

这个修复适用于模板类似 `{{fileName}}-{{heading}}-{{index}}`、`{{note}}-{{heading}}-图{{index}}` 的场景。
| `{{DATE:YYYYMMDD}}` | 当前日期，格式由 Moment 解析 |
| `{{frontmatter:key}}` | 当前笔记 YAML 字段 |
| `{{imageNameKey}}` | 当前笔记 YAML 中的 `imageNameKey` 字段 |

推荐模板：

```text
{{fileName}}-{{heading}}-图{{index}}
```

示例：当前笔记为 `碳点-激发依赖性.md`，当前标题为 `表面态机制`，则图片名可能生成：

```text
碳点-激发依赖性-表面态机制-图1.png
```

该机制解决的是图片脱离笔记语境后难以识别的问题。

### 5. 语义保留式图片重排

命令：

```text
按当前文章顺序重排图片编号（预览）
```

设置项：

```text
重排命名模式
```

可选：

| 模式 | 结果 |
|---|---|
| 保留语义 | `笔记名-1-原短名.png` |
| 纯编号 | `笔记名-1.png` |

推荐使用 **保留语义**。它能让图片既保留正文顺序，又保留原文件名中的语义信息。

示例：

```text
表面态机制.png
缺陷态示意图.png
```

重排后：

```text
碳点激发依赖性-1-表面态机制.png
碳点激发依赖性-02-缺陷态示意图.png
```

> 系统性风险：重排会触发文件重命名，并依赖 Obsidian 自动更新 Markdown 链接。插件会写入操作日志，但仍建议先使用预览入口。

### 6. 当前笔记图片归档到同名附件文件夹

命令：

```text
归档当前笔记图片到同名附件文件夹（预览）
```

默认会把当前笔记引用的图片移动到：

```text
当前笔记名.assets/
```

示例：当前笔记为：

```text
碳点激发依赖性.md
```

归档目标为：

```text
碳点激发依赖性.assets/
  碳点激发依赖性-1-表面态机制.png
  碳点激发依赖性-02-缺陷态示意图.png
```

可在设置页修改归档文件夹模板：

```text
{{fileName}}.assets
```

> 系统性风险：归档同时涉及“移动文件 + 重命名文件 + 更新正文链接”。这是高风险写操作。请优先使用预览，并确认没有目标路径冲突。

### 7. 安全图片链接清洗

命令：

```text
预览图片文件名清洗
应用上次图片文件名清洗结果
打开图片清洗侧栏
```

该功能用于把路径型图片链接收缩为文件名链接。

例如：

```md
![[assets/images/a.png|500]]
```

安全条件满足时可清洗为：

```md
![[a.png|500]]
```

为了避免同名图片导致误指向，建议保持：

```text
要求文件名唯一：开启
```

插件会在预览中显示：

- 将修改哪些笔记；
- 每处链接的 Before / After；
- 被跳过的链接及原因；
- 是否存在同名风险、外部链接、无法解析路径。

## 操作日志与撤销

插件会为以下操作写入日志：

- 当前行图片设置尺寸；
- 当前行图片重命名；
- 当前笔记图片尺寸统一；
- 当前文章图片编号重排；
- 当前笔记图片归档；
- 图片链接清洗。

日志目录：

```text
.obsidian/plugins/image-workflow/logs/
```

其中：

```text
last-operation.json
```

用于撤销上一次操作。

撤销命令：

```text
撤销上一次图片工作流操作
```

撤销规则：

- 如果文件仍在新路径，且旧路径未被占用，则尝试改回旧路径；
- 如果笔记正文仍等于日志中的 `afterContent`，则恢复为 `beforeContent`；
- 如果笔记已经被用户继续修改，则跳过该笔记，避免覆盖新内容。

> 边界：撤销不是版本控制系统。它只能回滚最近一次插件记录的操作，且要求相关文件和正文状态没有被其他操作破坏。

## 推荐设置

### 保守写作模式

适合正式库、重要论文笔记、长期项目库。

```text
自动重命名：关闭
粘贴后询问尺寸：开启
要求文件名唯一：开启
最终确认：开启
重排命名模式：保留语义
尺寸预设：300,400,500,600,800
```

### 日常写作模式

适合普通知识库。

```text
自动重命名：关闭
粘贴后询问尺寸：开启
要求文件名唯一：开启
最终确认：开启
重排命名模式：保留语义
图片命名模板：{{fileName}}-{{heading}}-图{{index}}
```

### 高自动化模式

适合测试库或临时库，不建议直接用于正式库。

```text
自动重命名：开启
粘贴后询问尺寸：关闭
要求文件名唯一：关闭
最终确认：关闭
```

## 建议工作流

### 写作时

1. 粘贴图片；
2. 在弹窗中确认图片名；
3. 点击尺寸预设，例如 `500`；
4. 继续写作。

### 整理当前笔记时

1. 执行 `检查当前笔记图片状态`；
2. 如果存在无尺寸图片，执行 `统一当前笔记图片尺寸`；
3. 如果图片名混乱，执行 `按当前文章顺序重排图片编号（预览）`；
4. 如果图片散落，执行 `归档当前笔记图片到同名附件文件夹（预览）`；
5. 如需缩短路径，执行 `预览图片文件名清洗`。

### 批量清洗前

1. 保持 `要求文件名唯一` 开启；
2. 执行 `预览图片文件名清洗`；
3. 查看跳过原因和差异；
4. 确认后再执行 `应用上次图片文件名清洗结果`。

## 已知边界

1. **Obsidian 链接更新是异步的**  
   文件重命名后，Obsidian 需要一点时间更新正文链接。插件加入了短暂等待与校验，但大库或同步盘环境仍可能延迟。

2. **同名图片会提高误清洗风险**  
   如果仓库中存在多个 `image.png`，短链接 `![[image.png]]` 可能不稳定。建议保持唯一性检查。

3. **Markdown 图片语法尺寸不是 Obsidian 标准宽度语法**  
   对 `![](a.png)`，插件会把尺寸写入 alt 文本，例如 `![500](a.png)`。Obsidian 对 Wiki 嵌入 `![[a.png|500]]` 的支持更稳定。

4. **归档功能属于高风险写操作**  
   它会移动文件并更新链接。正式库中建议先备份，或至少先用预览确认。

5. **撤销不等于完整历史版本**  
   如果操作后又手动编辑了同一篇笔记，撤销会跳过，避免覆盖新编辑。

## 安装

将插件文件夹放入：

```text
.obsidian/plugins/image-workflow/
```

文件夹内应至少包含：

```text
main.js
manifest.json
styles.css
README.md
```

然后在 Obsidian 中进入：

```text
设置 → 第三方插件 → 已安装插件 → 启用“图片重命名与链接清洗”
```

### 8. 图片尺寸写入稳定性修复

本版针对“选择图片尺寸后没有写入”的问题做了专项修复。

修复点：

- 尺寸写入不再只依赖打开弹窗时记录的旧字符位置；
- 如果 Obsidian 在重命名后自动刷新了链接，插件会重新扫描当前行、附近行和当前笔记正文，重新定位目标图片链接；
- 文本替换优先使用 Obsidian 编辑器的 `replaceRange`，并在写入后校验正文是否真的包含新尺寸后缀；
- 写入失败时会显示明确提示，而不是静默失败。

推荐验证流程：

```text
粘贴图片 → 选择或输入尺寸 500 → 确认 → 检查正文是否变为 ![[图片名.png|500]]
```

如果图片已经存在于当前行，也可以使用：

```text
命令面板 → 当前行图片设置尺寸
```

该命令只修改尺寸，不重命名文件，适合排查尺寸写入问题。


## 2026-06-20 粘贴图片尺寸写入二次修复

### 问题现象

执行“粘贴图片 → 选择尺寸 500 → 确认”后，正文仍然只出现：

```md
![[图片名.png]]
```

而没有写入：

```md
![[图片名.png|500]]
```

### 根因

Obsidian 粘贴图片时，文件创建、正文链接插入、重命名后的链接刷新并不是严格同步完成。旧版本在确认弹窗后立即查找当前行链接；如果当时正文链接尚未插入，或者刚被 Obsidian 自动刷新，插件就找不到可替换的链接位置，导致尺寸写入被跳过。

### 本版修复

本版新增了强制尺寸写入链路：

1. 重命名后会在当前笔记全文中反复查找指向目标图片的嵌入链接；
2. 不再只依赖当前光标行，也不只依赖弹窗打开时记录的字符位置；
3. 支持在链接稍后才出现时继续等待并写入尺寸；
4. 支持通过文件名、完整路径、短路径三种方式匹配目标图片；
5. 写入后会再次校验正文中是否已经出现 `|500` 这类尺寸后缀。

### 推荐验证

安装后执行：

```text
粘贴图片 → 选择尺寸 500 → 确认
```

预期结果：

```md
![[图片名.png|500]]
```

如果仍未写入，优先检查“粘贴时弹出尺寸输入框”是否开启，以及是否有其他插件在粘贴后立即改写图片链接。

## 2026-06-20 直接重写尺寸链路修复

本版将“粘贴图片 → 重命名 → 选择尺寸 → 确认”的尺寸写入逻辑改为更直接的链路：

1. 粘贴图片后，先记录当前 Markdown 中刚插入图片链接所在行与字符区间；
2. 用户确认新文件名和尺寸后，先执行图片文件重命名；
3. 重命名完成后，不再等待 Obsidian 自动链接刷新结果作为主路径，而是直接把记录位置的图片嵌入重写为：

```md
![[图片名.png|500]]
```

如果原记录位置已经被 Obsidian 或其他插件刷新，插件会回退到当前行/全文查找对应图片链接，再执行同样的直接重写。

该修复针对的典型问题是：选择了 `500`，但正文仍然停留在：

```md
![[图片名.png]]
```



## 链接写入格式说明

图片写入使用 Obsidian 的图片嵌入格式 `![[图片.png]]`，带尺寸时为 `![[图片.png|500]]`。插件不会写成普通 Wiki 链接 `[[图片.png]]`。

## 修复：当前文章图片重排命名链式膨胀

本版修复了“当前文章图片重排预览”中出现的链式重复命名问题。

### 问题表现

旧逻辑在命名模板已经包含 `{{index}}` 时，仍然先生成一个带序号的基础名，再追加重排序号和原文件名，容易出现：

```md
![[张三-测试123-2-1-张三-测试1-1.png]]
```

这类结果属于错误命名。正确逻辑应由模板直接决定最终文件名。

### 新逻辑

如果图片命名模板包含 `{{index}}`，重排时直接按正文顺序把 `{{index}}` 替换为当前序号：

```text
{{fileName}}-{{heading}}-{{index}}
```

会生成：

```md
![[张三-测试1-1.png]]
![[张三-测试1-2.png]]
```

不会再追加旧文件名，也不会生成 `张三-测试1-1-1.png` 或 `张三-测试123-2-1-旧名.png`。

### 规则边界

- 模板包含 `{{index}}`：模板即完整命名规则，重排只替换序号。
- 模板不包含 `{{index}}`：插件才会使用“基础名 + 序号”的重排规则。
- “保留语义”只在模板不含 `{{index}}` 时追加旧文件名，避免链式命名膨胀。


## 修复：重排链路自查与代码清洗

本版对“当前文章图片重排”做了自查式修复，目标是避免再次出现 `张三-测试123-2-1-张三-测试1-1.png` 这类链式膨胀文件名。

### 修复点

1. **重排顺序改为读取当前笔记正文中的图片出现顺序**

   旧逻辑主要依赖 metadata cache 中的 embed 列表。新逻辑在构建重排计划时直接读取当前 Markdown 正文，解析 `![[...]]`、`[[...]]` 和 `![](...)`，再按正文出现顺序生成任务。

2. **模板包含 `{{index}}` 时禁止追加旧文件名**

   如果命名模板是：

   ```text
   {{fileName}}-{{heading}}-{{index}}
   ```

   重排只会替换 `{{index}}`，不会再叠加“重排序号”或“原文件名”。

3. **修复重复目标名检测**

   旧逻辑使用 `Set` 后再统计重复目标路径，重复项已经被去重，导致“目标文件名重复”检测失效。新逻辑改为数组计数，能正确阻止多个图片被重排或归档到同一路径。

4. **重排 / 归档后强制规范化图片嵌入链接**

   文件重命名完成后，插件会再次扫描当前笔记，把相关图片链接统一重写为图片嵌入格式：

   ```md
   ![[图片名.png]]
   ![[图片名.png|500]]
   ```

   这会修复 Obsidian 自动更新后残留的普通 Wiki 链接：

   ```md
   [[图片名.png]]
   ```

5. **修复序号递增正则**

   修复 `nextIndexedNameFromStem()` 中 `\d+` 转义错误，避免模板序号递增判断失效。

6. **修复空名判断正则**

   修复 `meaninglessRegex` 中空白字符 `\s` 转义错误，避免某些模板被错误判断为有效或无效。

### 当前推荐模板

```text
{{fileName}}-{{heading}}-{{index}}
```

编号位数建议保持：

```text
1
```

这样连续图片应生成：

```md
![[张三-测试1-1.png]]
![[张三-测试1-2.png]]
![[张三-测试1-3.png]]
```

如果选择尺寸 `500`，应生成：

```md
![[张三-测试1-1.png|500]]
![[张三-测试1-2.png|500]]
![[张三-测试1-3.png|500]]
```

## 代码重构与清洗记录（1.2.2）

本版主要处理内部结构，不改变插件的使用边界：仍然只围绕 Markdown 图片嵌入链接工作。

### 重构点

1. **统一图片链接解析入口**

   新增统一解析函数：

   ```js
   createImageLinkRegex()
   parseImageLinkMatch()
   makeImageLinkItemFromMatch()
   ```

   原先多个功能各自维护图片链接正则，容易出现“粘贴可识别、重排不可识别、体检又是另一套识别规则”的问题。现在当前行识别、全文扫描、体检、重排、归档等链路逐步收敛到同一套解析结构。

2. **清理重复字段语义**

   图片链接解析结果中明确保留：

   - `lineNumber`：行号，用于编辑器定位；
   - `lineText`：原始行文本，用于调试和后续扩展；
   - `line`：兼容旧 UI 显示逻辑，仍然保留为行号。

   这样避免旧代码里 `line` 同时表示“行文本”和“行号”的语义混乱。

3. **修复全局正则状态风险**

   所有逐行扫描现在按行创建新的 global regex，避免 JavaScript `RegExp.lastIndex` 在多行复用时造成漏匹配。

4. **删除无入口冗余方法**

   移除未被调用的 `getCurrentLineFirstEmbedFile()`，减少维护噪声。

5. **保留高风险链路的日志与回滚**

   本次没有削弱重命名、重排、归档的日志能力；涉及文件移动或正文改写的功能仍会写入 `.obsidian/plugins/image-workflow/logs/last-operation.json`。

### 自查结果

已执行：

```bash
node --check main.js
```

语法检查通过。

### 仍然保留的边界

插件没有改成多文件工程，也没有引入构建工具。原因是 Obsidian 社区插件在手动安装时通常依赖 `main.js + manifest.json + styles.css` 的简单结构；本版优先做**低迁移成本重构**，避免引入打包链导致新的安装风险。

## 1.2.4 重排命名规则修复

### 修复点

- 当前文章图片重排时，图片名现在会严格按当前命名模板重新生成。
- `{{heading}}` 在重排时不再取当前光标所在标题，而是取每张图片在正文中所在位置的上级标题。
- 标题改名后，重新执行“当前文章图片重排”会按新标题生成新图片名。
- 当旧文件名已经明显是插件生成名时，重排不会再把旧图片名拼接到新文件名后面，避免出现：

```md
![[张三-1-张三-测试1-1.png]]
```

应生成：

```md
![[张三-新标题-1.png]]
![[张三-新标题-2.png]]
```

### 机制说明

重排命名现在遵循：

```text
读取当前 Markdown 正文
→ 按正文顺序解析图片链接
→ 根据每张图片所在行向上查找最近标题
→ 用当前命名模板渲染 {{fileName}} / {{heading}} / {{index}}
→ 生成目标图片名
```

因此，如果你修改了标题，只需要重新执行重排预览，预览中的新文件名就应跟随新标题变化。
